## 原理:virtio-mmio 设备发现是怎么工作的

核心思想:**QEMU 在固定的物理地址放了一排"插槽",每个插槽可能插着一个 virtio 设备。内核挨个插槽去敲门,问"你是 virtio 设备吗?是什么类型?"**

具体到你的代码:

**三个常量定义了"插槽阵列"的布局:**

```rust
VIRTIO_MMIO_BASE   = 0x1000_1000   // 第一个插槽的物理地址
VIRTIO_MMIO_STRIDE = 0x1000        // 每个插槽间隔 0x1000(4KB)
VIRTIO_MMIO_COUNT  = 8             // 一共 8 个插槽
```

这是 **QEMU `virt` 机器的硬约定**——QEMU 就是把 virtio-mmio 设备放在 `0x1000_1000` 开始、每隔 `0x1000` 一个、共 8 个这些固定地址上。内核不需要"扫描整个内存找设备",而是**知道该去哪几个地址找**(因为 QEMU 文档/设备树规定了)。8 个插槽的地址就是:

```
slot 0: 0x1000_1000
slot 1: 0x1000_2000
slot 2: 0x1000_3000
...
slot 7: 0x1000_8000
```

**每个插槽是一组内存映射寄存器**(MMIO = Memory-Mapped IO)。访问 `0x1000_1000` 这个物理地址,不是访问内存,而是访问"插槽 0 那个设备的寄存器"。

**循环里"敲门"的过程:**

```rust
for i in 0..8 {
    phys_base = BASE + i * STRIDE;        // 算出第 i 个插槽地址
    virt_base = phys_to_virt(phys_base);  // 转成内核能访问的虚拟地址
    hdr = VirtioMmioHeader::new(virt_base); // 把这块地址当成 virtio 寄存器头

    magic = hdr.magic();        // 读 MagicValue 寄存器(偏移 0)
    if magic != VIRTIO_MAGIC {  // 0x74726976 = "virt" 的 ASCII
        continue;               // 不是 virtio 设备(插槽空的),下一个
    }
    device_id = hdr.device_id();              // 读 DeviceID 寄存器
    if device_id == VIRTIO_DEVICE_ID_BLOCK {  // 2 = block device
        return Some(phys_base);               // 找到块设备!返回它的地址
    }
}
```

**两道检查:**

1. **MagicValue**:每个 virtio 设备的第一个寄存器固定是 `0x74726976`(小端的 "virt")。读这个就知道"这个插槽有没有插 virtio 设备"。空插槽读出来是 0 或乱值,`!= VIRTIO_MAGIC` 就跳过。这是"敲门确认是不是 virtio"。
2. **DeviceID**:确认是 virtio 后,读 DeviceID 寄存器看**是什么类型**的 virtio 设备。`2` = 块设备,`1` = 网卡,等等。你只要块设备,所以 `device_id == 2` 才要。

**为什么是"内存映射"**:`hdr.magic()` 内部其实是 `read_volatile(virt_base + 0)` 之类——读那个物理地址,QEMU 拦截这个读,返回设备的 MagicValue。整个设备的控制接口(magic、version、device_id、features、queue 配置...)都是这块地址上不同偏移的寄存器。这就是 "virtio-mmio" 的 "mmio" 含义。

**对比 loongarch 的 virtio-pci**:loongarch 没有这种"固定地址插槽阵列",设备挂在 PCI 总线上,要先**枚举 PCI 总线**(扫 bus/device/function)找到 virtio-blk,再从 PCI 配置空间拿到设备地址。所以 loongarch 那边要 `scan_pci_bus` + `find_virtio_blk_pci`——多一层 PCI 枚举。但找到设备之后,virtqueue 和 device 那两层完全一样。这就是为什么 riscv 的发现简单(地址固定)、loongarch 复杂(要枚举 PCI)。



## 关键:QEMU 里盘的顺序

有个要注意的:**两块盘在哪个 slot,取决于 QEMU 命令行 `-device` 的顺序。** QEMU 通常按你 `-device virtio-blk-device` 的出现顺序,从 slot 0(其实是高地址往低排,QEMU virt 的 virtio-mmio 顺序有时是反的)依次分配。**所以你不能假设"slot 0 一定是 ext4"**——要么靠顺序约定,要么读每块盘的内容判断(比如读 ext4 magic 区分哪块是 ext4)。

**最稳的做法**:probe 返回所有盘后,**读每块盘判断类型**——哪块有 ext4 magic(sector 2 offset 56 = 0xef53)就是 rootfs,另一块就是 FAT。这样不依赖 slot 顺序。或者简单点:**约定第一块(probe 返回列表的 [0])是 ext4、第二块 [1] 是 FAT**,然后在 QEMU 命令行里按这个顺序挂——能用,但依赖顺序。

## 问题一:能有八块盘吗?

**理论上是的**,QEMU virt 机器的 virtio-mmio 有 8 个插槽,你可以挂最多 8 个 virtio 设备。但有几个现实约束:

**这 8 个插槽是给"所有 virtio 设备"共享的,不只是块设备。** 如果你以后加 virtio-net(网卡)、virtio-rng 等,它们也占插槽。所以不是"8 块盘",是"8 个 virtio 设备总共"。

**而且不一定真有 8 个能用。** QEMU 提供 8 个插槽地址,但只有你在命令行 `-device` 挂了的才真的有设备,没挂的插槽是空的(magic 读出来不对,probe 跳过)。所以实际能用几块,取决于你 QEMU 挂了几个。

## 问题二(关键):"就找到个地址,0x1000 大小,能干什么?"

这是个**非常好的疑问**,说明你在认真想 MMIO 到底是什么。答案会颠覆你的直觉:

**那 `0x1000`(4KB)不是"数据空间",是"控制面板"。** 它装不下一块盘的数据,但它从来就不是用来装数据的。

### 那 4KB 里是什么?

那 4KB 是设备的**寄存器**——一排"旋钮和仪表"。每个偏移是一个控制寄存器:

```
偏移 0x000: MagicValue    (读:确认是 virtio)
偏移 0x004: Version
偏移 0x008: DeviceID       (读:设备类型)
偏移 0x00c: VendorID
偏移 0x010: DeviceFeatures (读:设备支持什么)
偏移 0x020: DriverFeatures (写:驱动启用什么)
偏移 0x030: QueueSel       (写:选哪个队列)
偏移 0x038: QueueNumMax
偏移 0x044: QueueReady
偏移 0x050: QueueNotify    (写:通知设备"有新请求了")
偏移 0x060: InterruptStatus
偏移 0x070: Status
偏移 0x080: QueueDescLow    (写:告诉设备描述符表的物理地址)
...
偏移 0x100+: 设备特定配置(blk 的容量等)
```

**这 4KB 全是"控制寄存器",不存任何盘数据。** 它是你和设备对话的**接口**,不是仓库。

### 那盘的数据(几个 GB)在哪?

**不在这 4KB 里。** 这是 MMIO 的精妙处:**数据根本不经过这块地址。**

回想你 `read_sector_sync` 干的事:

```
1. 你在自己的 DMA 内存(另一块物理页)准备好:请求头、数据缓冲、状态字节
2. 你把这些缓冲的【物理地址】写进设备寄存器(QueueDesc 等)
3. 你写 QueueNotify 寄存器,通知设备"我放了个请求"
4. 设备(QEMU)自己去读你那块 DMA 内存,知道你要读 sector N
5. 设备把磁盘 sector N 的 512 字节,【直接写进你的 DMA 数据缓冲】
6. 设备写 used ring,你轮询到完成
7. 你从【你自己的 DMA 缓冲】读出那 512 字节
```

**看出来了吗?** 数据(512 字节)是设备**直接搬进你的 DMA 内存**的,**不经过那 4KB 寄存器区**。那 4KB 只是用来:

- 告诉设备"我的请求队列在哪个物理地址"(给地址)
- 戳一下"有新请求了"(QueueNotify)

**真正的数据传输走的是 DMA**(Direct Memory Access):设备直接读写你指定的物理内存。这就是为什么 4KB 够用——它只传"指针和信号",不传数据本身。数据在你的 DMA 页里(你代码里的 `VirtioBlkDma`,那是另一块 alloc 的物理页)。

### 一个比喻

那 4KB 寄存器区像**餐厅的服务铃 + 点餐单夹**:

- 你不在这块小地方做菜(数据不在这)
- 你在点餐单(DMA 内存)写"要 sector N"
- 把单子的位置告诉服务员(写 QueueDesc 寄存器 = 给地址)
- 按铃(写 QueueNotify = 通知)
- 厨房(设备)去你桌上拿单、把菜(数据)直接端到你桌上(DMA 缓冲)
- 你从自己桌上(DMA 缓冲)取菜

**服务铃只要巴掌大(4KB),因为菜不经过铃。** 菜(数据)直接在你和厨房之间端来端去(DMA)。

### 所以"找到个地址"能干什么?

那个地址(`phys_base`)是**这块盘的控制面板入口**。拿到它,你就能:

- 读它的寄存器:确认是块设备、读容量、读支持的 features
- 写它的寄存器:配置队列、启用 features、通知请求
- 通过它驱动整块盘的所有读写(数据走 DMA,但**指挥**走这些寄存器)

**一个地址 = 一块盘的完整控制权。** 你 `init_from_phys_base(phys_base)` 就是拿这个地址,把整块盘驱动起来。两块盘 = 两个 `phys_base` = 两套控制面板 = 两块盘各自独立驱动。

## 这对你 FAT 的意义

你 probe 找到两个地址(两块盘的控制面板),分别 `init_from_phys_base`,得到两个 `VirtioBlkDevice`。每个能独立读写(各自的寄存器 + 各自的 DMA)。一块给 ext4,一块给 FAT。**它们互不干扰,因为是两套独立的控制面板 + 两块独立的物理盘。**

## 一个你之后会遇到的细节

既然两块盘各有自己的 DMA 区——你现在的 `VirtioBlkDma` 是每个 `VirtioBlkDevice` 实例自己 alloc 的(`VirtioBlkDma::new()` 在 init 里调)。**所以两个设备实例自动有各自的 DMA,不会冲突。** 你的代码结构已经天然支持多盘了(DMA 在 inner 里,每个实例一份),这是 GPT 那部分写得对的地方。

## 先纠正一个小误解:那次"沟通"没有访问硬盘

你说"多进行了一次硬盘内存沟通"——其实**设备读你的 DMA 内存,不是访问硬盘,是访问主存(RAM)**。设备(QEMU 模拟的 virtio-blk)读你那块 DMA 页,走的是**内存总线**,不碰物理硬盘。真正的硬盘访问只在第 5 步(设备把 sector N 的数据搬进来)。所以你担心的"多一次硬盘沟通"不存在——多的是"设备读一次主存里的请求描述符",而读主存是很快的(相对硬盘)。

但你的核心疑问仍然成立:**为什么不把 sector N 直接写进寄存器,省掉"设备读主存请求"这一步?** 这才是关键。

## 简单情况下,你的想法确实可行

如果 virtio-blk 只支持"一次读一个 512 字节扇区",那确实可以设计成:

```
寄存器偏移 0x100: 写入 sector 号
寄存器偏移 0x108: 写入数据缓冲地址
寄存器偏移 0x110: 写 1 触发读
```

这种设计**真实存在**——很多简单的老式硬件(IDE/ATA 磁盘控制器)就是这么干的:往几个寄存器写 LBA 地址、扇区数、命令,然后读状态寄存器。**对单个简单请求,寄存器直传确实更省。**

所以你的直觉对**简单场景**是对的。virtio 用 DMA 间接,是为了**复杂场景**。下面讲为什么。

## 为什么 virtio 选 DMA 描述符,而非寄存器直传

### 理由 1:批量 / 异步 —— 一次提交多个请求

寄存器直传的根本限制:**寄存器只有一组**。你写 sector 到 `0x100`,再写一个 sector 进去,就**覆盖**了前一个。所以寄存器直传天然是"一次一个请求,必须等它完成才能发下一个"——**同步、串行**。

而 DMA 描述符环(virtqueue)能做到:

```
你在 avail ring 里一次塞 10 个请求(10 条描述符链)
按一次 QueueNotify
设备并发处理这 10 个,完成一个往 used ring 放一个
你异步收割
```

**寄存器装不下"10 个待处理请求的队列",但内存里的环形队列可以。** 现代磁盘(尤其 SSD、NVMe)的性能关键就是**深队列 + 并发**——同时提交几十上百个请求,让设备内部调度、并行。寄存器直传做不到这个,DMA 队列是为高吞吐而生。

你现在的实现是"一次一个、同步等待"(`read_sector_sync`),所以你**暂时没用到**这个优势——你的用法其实接近寄存器直传的简单模型。但 virtio 的设计是为了**能扩展到高并发**,所以用了队列。

### 理由 2:可变长 / 复杂请求 —— 寄存器放不下

一个请求可能很复杂:

- 读多个不连续的扇区(scatter-gather)
- 一个请求的数据要分散到内存里好几块不连续的缓冲区
- 不同设备类型(net 的包、scsi 的命令)请求结构完全不同

**描述符链(desc chain)能表达任意复杂的"这个请求由哪几段内存组成"**:

```
desc0 → desc1 → desc2 → ...   (链表,想多长多长)
```

你的 blk 请求是 3 段(头+数据+状态),但 net 可能是几十段(一个网络包分散在多个缓冲)。**寄存器是固定几个,表达不了这种可变长、多段的结构。** 内存里的链表可以。

### 理由 3:统一框架 —— 所有 virtio 设备共用一套机制

virtio 要支持 blk、net、console、gpu、rng... 几十种设备。如果每种设备都用"自己的一组寄存器直传",那每种设备的驱动、每种 host 实现都不同,无法统一。

**virtio 的杀手锏:所有设备共用同一套 virtqueue 机制**(描述符环),区别只在"描述符指向的缓冲里放什么"。你写的 queue 代码,virtio-net 几乎能照用。这就是为什么我之前说"第 2 层 virtqueue,blk 和 net 完全一样"。**寄存器直传无法统一**(每种设备寄存器布局不同),DMA 描述符环可以(机制统一,内容自定义)。

### 理由 4:寄存器访问(MMIO)其实很贵

反直觉的一点:**写 MMIO 寄存器,在真实硬件 / 虚拟化里,比写普通内存慢得多。**

- 写普通 RAM:CPU 缓存,几纳秒
- 写 MMIO 寄存器:不能缓存,要走总线到设备,可能几百纳秒;在虚拟化里更要 **VM exit**(陷出到 hypervisor),代价上千周期

所以如果一个请求要写 5 个寄存器(sector、地址、长度、命令...),那是 **5 次昂贵的 MMIO 写**。而 virtio 的做法:

```
在内存里(便宜)填好整个描述符链
只写 1 次 MMIO 寄存器(QueueNotify)通知设备
```

**把"准备请求"全放在便宜的内存操作里,只用 1 次昂贵的 MMIO 戳一下。** 请求越复杂,这个优势越大——寄存器直传要 N 次 MMIO 写,virtio 永远只要 1 次 notify。

**你担心的"多一次设备读内存",其实换来了"少好几次昂贵的 MMIO 寄存器写"。** 设备读内存(DMA)对设备是便宜的,而 CPU 写 MMIO 是贵的。这个交换是划算的。

## 总结这个权衡

|                  | 寄存器直传           | DMA 描述符环(virtio) |
| ---------------- | -------------------- | -------------------- |
| 单个简单请求     | 更直接               | 多一次设备读内存     |
| 批量/并发请求    | 做不到(寄存器会覆盖) | 队列天然支持深度并发 |
| 复杂/可变长请求  | 寄存器放不下         | 链表任意表达         |
| 多种设备统一     | 每种设备不同寄存器   | 统一机制             |
| CPU 侧 MMIO 开销 | N 次昂贵写           | 1 次 notify          |

**你的直觉对"单个简单同步请求"是对的**——那种情况寄存器直传确实更省一次内存读。virtio 选 DMA 是为了**并发、可变长、统一框架、减少昂贵的 MMIO 写**,这些在高性能、多设备场景下完胜,代价是单个简单请求多一次(便宜的)设备读内存。

## 一个精准的洞察

你其实发现了一个真实的事实:**对你当前的用法(一次一个、同步、固定 3 段),virtio 的 DMA 机制确实"杀鸡用牛刀"了。** 你完全可以用更简单的寄存器直传实现一个 blk 驱动。

但 virtio 不是为"教学内核一次读一个扇区"设计的,它是为"生产环境高并发 IO"设计的工业标准。你用它,是因为:

1. QEMU 提供的就是 virtio(你得按它的协议来)
2. 学会 virtio = 学会真实世界的设备驱动范式(DMA + 描述符环,NVMe、现代网卡都是这套)

## 描述符本身只描述"内存",不描述"角色"

你观察得对:一个描述符只有四个字段:

```
addr   这段内存在哪
len    多长
flags  设备是读这段还是写这段(WRITE = 设备写)
next   下一段是谁
```

**它确实没有"我是请求头 / 我是数据 / 我是状态"这种字段。** 描述符是通用的,virtio-net、virtio-gpu 都用同一个描述符结构,它只描述"一段内存 + 方向",不带任何设备语义。

那设备怎么知道角色?——**靠"这是个 virtio-blk 设备"这个上下文 + 固定的顺序约定。**

## virtio-blk 的协议规定:一个请求必须按这个顺序

virtio 规范里,**virtio-blk 设备**的每个请求,描述符链**必须**是这个固定结构:

```
第 1 段(链头):  请求头 struct virtio_blk_req 的前 16 字节
                 (type + reserved + sector),设备【读】
第 2..n 段:      数据缓冲,读操作设备【写】/ 写操作设备【读】
最后 1 段:       1 字节状态,设备【写】
```

**这是 virtio-blk 这个设备类型的铁律。** 设备拿到一条描述符链,它**不需要猜**每段是什么——因为协议规定死了:

- **链的第一段,一定是请求头**(16 字节的 type+sector)
- **链的最后一段,一定是 1 字节状态**
- **中间的,一定是数据**

所以设备的逻辑是:

```
拿到链头 desc[0] → "按 virtio-blk 协议,链头必是请求头" → 读它的 16 字节,解析 type 和 sector
顺着 next 走到中间段 → "这些是数据" → 按 type 决定读还是写
走到最后一段 → "最后一段必是状态" → 写 1 字节完成码
```

**设备是靠"位置"(第一个/中间/最后)+"我是 blk 设备"来确定角色的,不是靠描述符里的字段。**

## 回答你的两个具体疑问

### 疑问 1:desc[2] 为什么不被理解成"数据缓冲"?

```
desc[2] = { addr: 0x8050_0400, len: 1, flags: WRITE, next: 0 }
```

设备不会把它当数据,因为:

1. **它是链的最后一段**(`next: 0`,无下一段)。virtio-blk 协议规定**最后一段必是状态字节**。
2. **而且它只有 1 字节**(`len: 1`)。状态就是 1 字节,数据是 512 的倍数。

设备顺着链走:desc[0](https://claude.ai/chat/头)→ desc[1](https://claude.ai/chat/next=2)→ desc[2](https://claude.ai/chat/next=0,到底了)。**走到 next=0 的那一段,就是状态段。** 所以 desc[2] 被理解成状态,不是数据——**因为它在"最后一段"这个位置。**

### 疑问 2:desc[0] 为什么是请求头,不是数据?

```
desc[0] = { addr: 0x8050_0000, len: 16, flags: NEXT, next: 1 }
```

设备从 avail 环拿到链头索引是 0,**desc[0] 就是链头(第一段)**。virtio-blk 协议规定**第一段必是请求头**。所以设备:

1. 拿到链头 desc[0]
2. "按协议,链头是请求头" → 读它的 16 字节
3. 解析:前 4 字节 type、后 8 字节 sector → 知道是读还是写、读哪个扇区

**设备不会把 desc[0] 当数据缓冲**,因为协议规定第一段是头。而且注意 desc[0] **没有 WRITE flag**(`flags: NEXT`,没有 WRITE)——这也对:请求头是"设备读"(读你的指令),所以无 WRITE。如果设备把它当成"往这写数据",和无 WRITE flag 就矛盾了。

## 那 flags 的 WRITE 到底起什么作用?

你可能会问:既然角色靠顺序定,那 flags 的 WRITE 还有什么用?

**WRITE flag 不决定"角色",它决定"数据方向"。** 它告诉设备:**这一段,你是该读它还是写它。** 这是给设备做 DMA 时用的——设备要知道"这段内存我是读出来,还是写进去"。

具体:

```
读操作(读磁盘):
  desc[0] 请求头   无 WRITE  → 设备读(读你的指令)
  desc[1] 数据     有 WRITE  → 设备写(把磁盘数据写进来)
  desc[2] 状态     有 WRITE  → 设备写(写完成码)

写操作(写磁盘):
  desc[0] 请求头   无 WRITE  → 设备读(读你的指令)
  desc[1] 数据     无 WRITE  → 设备读(读你要写的数据)  ← 这里和读不同!
  desc[2] 状态     有 WRITE  → 设备写(写完成码)
```

**角色(头/数据/状态)= 看位置。方向(设备读/写)= 看 WRITE flag。** 两个信息是分开的:

- 位置告诉设备"这段是干什么的"
- WRITE flag 告诉设备"这段我该往哪个方向 DMA"

对数据段,**同样是"数据"这个角色,但读操作要 WRITE(设备写入)、写操作不要 WRITE(设备读取)**——这就是你做写路径时改 desc[1] flag 的原因。角色没变(都是数据),方向变了(读 vs 写)。

## 一个验证:如果你搞错了会怎样

假设你写磁盘时,desc[1] **错误地**带了 WRITE flag(本该不带):

- 设备看 desc[1] 有 WRITE → 以为"我该往这段写"
- 但写磁盘操作里,这段本该是"设备读你的数据"
- 设备困惑/出错:它可能往你的数据缓冲写垃圾,而不是读你的数据去写磁盘
- 结果:磁盘没写对,或者设备报错

**所以 WRITE flag 必须和操作方向匹配**,这是你写路径最容易错的地方(我之前强调过)。但**角色(头/数据/状态)永远靠位置,你不用操心**——只要保持"头在前、数据在中、状态在最后"这个顺序就行。

```
描述符里有的:    addr(在哪)+ len(多长)+ flags(设备读/写这段)+ next(下一段)

角色怎么定:      靠 virtio-blk 协议的固定顺序——
                第一段=请求头,最后一段=状态,中间=数据
                设备靠"位置 + 我是 blk 设备"推断角色,不靠字段

```

## 两个信息,两个层次,两个处理者

### `type=0/1`(在请求头里)—— 给 virtio-blk 设备逻辑用

请求头里的 `type` 是 **virtio-blk 这个设备的语义**:"我要对磁盘做什么操作"。

```
type=0 (VIRTIO_BLK_T_IN)  → 从磁盘读
type=1 (VIRTIO_BLK_T_OUT) → 写到磁盘
type=4 (VIRTIO_BLK_T_FLUSH) → 刷盘
type=11 (VIRTIO_BLK_T_DISCARD) → 丢弃
...
```

这是**磁盘操作的种类**,是 blk 设备的"业务逻辑"。设备读到 type,知道"哦,这次是要读磁盘 / 写磁盘 / 刷盘"。

### `WRITE flag`(在描述符里)—— 给 virtqueue 传输层用

描述符的 `WRITE flag` 是 **virtqueue 这个通用传输机制的语义**:"这段内存,设备做 DMA 时是往里写、还是从里读"。

```
有 WRITE  → 设备往这段内存【写】(device-writable)
无 WRITE  → 设备从这段内存【读】(device-readable)
```

这跟"磁盘操作"无关,它只描述**内存的 DMA 方向**。virtio-net、virtio-gpu 全用这个,跟磁盘没关系。

## 关键:它们在不同抽象层,互不知道对方

```
┌─────────────────────────────────────────┐
│  virtio-blk 设备逻辑层                      │
│  "我要读磁盘 sector 2"                      │
│  用 type 表达                              │ ← 请求头的 type
└─────────────────────────────────────────┘
              ↓ 需要把数据搬进/搬出内存
┌─────────────────────────────────────────┐
│  virtqueue 传输层(通用,所有 virtio 设备共用)│
│  "把这段内存当 DMA 目标,方向是写入"           │
│  用 WRITE flag 表达                        │ ← 描述符的 flag
└─────────────────────────────────────────┘
```

**上层(blk 逻辑)说"读磁盘",下层(virtqueue)说"往这段内存 DMA 写入"。** 它们是两个独立的层,各说各的语言。virtqueue 层根本不懂"磁盘""读写磁盘"这种概念——它只懂"内存段 + DMA 方向"。

## 为什么"读磁盘"时两者看起来重复了

"读磁盘"这个操作,拆解成两层:

```
blk 逻辑层:  type=0,"从磁盘读数据"
                ↓ 这数据要放哪?要放进 guest 的内存缓冲
virtqueue 层: 那段缓冲要标 WRITE,因为"设备要往它【写入】(把读到的磁盘数据放进去)"
```

**"从磁盘读" 在内存层面 = "往内存缓冲写入"。** 所以 type=0(读磁盘)对应数据段 WRITE(设备写内存)。看起来重复,其实是**同一操作在两层的不同投影**:

- 站在磁盘角度:读(type=0)
- 站在内存角度:写(设备把数据写进你的缓冲,WRITE)

它们**方向甚至是"相反"的措辞**——磁盘读 = 内存写。不是简单重复,是视角不同。

## 写磁盘时,你立刻看出它们不重复了

看写操作:

```
blk 逻辑层:   type=1,"写到磁盘"
                 ↓ 要写的数据从哪来?从 guest 的内存缓冲读
virtqueue 层:  那段缓冲【无 WRITE】,因为"设备要从它【读取】(读出数据去写磁盘)"
```

写磁盘:`type=1` + 数据段**无 WRITE**。

- 磁盘角度:写(type=1)
- 内存角度:读(设备从你的缓冲读出数据,无 WRITE)

**又是相反的措辞:磁盘写 = 内存读。**

## 一张表看清"两层 + 相反投影"

| 操作   | 请求头 type(磁盘视角) | 数据描述符 flag(内存DMA视角) |
| ------ | --------------------- | ---------------------------- |
| 读磁盘 | type=0(读磁盘)        | **有 WRITE**(设备写内存)     |
| 写磁盘 | type=1(写磁盘)        | **无 WRITE**(设备读内存)     |

**注意两列永远是"相反"的**:磁盘读↔内存写,磁盘写↔内存读。这正说明它们不是重复——一个讲磁盘操作,一个讲内存 DMA 方向,而磁盘操作和内存方向天然相反。

## 那为什么不省掉一个?

你可能想:既然 type=0 就够推出"数据段要 WRITE",设备自己推不行吗?干嘛让我标两次?

**几个原因:**

**1. 分层解耦。** virtqueue 传输层要保持**通用**——它服务 blk、net、gpu 所有设备。它不能内置"如果是 blk 且 type=0 就把第二段当 WRITE"这种 blk 专属逻辑。传输层只认"描述符 + WRITE flag",不认 type。**type 是 blk 私货,WRITE flag 是传输层通货。** 让传输层去解析 blk 的 type 会破坏分层。

**2. 设备做 DMA 前就要知道方向。** 设备在内存里 DMA 搬数据时,靠的是 WRITE flag 决定往哪搬。它处理描述符链是"通用流程"(对任何设备一样):看 flag 决定读/写每段。如果没有 WRITE flag、要靠解析每种设备的请求头来推方向,那每种设备的传输处理都不同了,失去统一性。

**3. 安全/校验。** 两个信息独立,设备可以**交叉校验**:如果 type=0(读磁盘)但数据段没标 WRITE,设备能发现"请求不合法"(读磁盘却不让我写内存,矛盾),拒绝它。两个独立的声明提供了冗余校验。

**4. 复杂请求里不是一一对应。** 简单 blk 请求里 type 和 flag 看着能互推,但复杂场景(scatter-gather、多段数据混合方向)里,描述符的 flag 提供了**逐段**的精确方向控制,不是请求头一个 type 能概括的。

## 一句话总结

```
type(请求头):  blk 设备的业务语义 —— "对磁盘做什么"(读/写/刷盘)
WRITE flag(描述符): virtqueue 的传输语义 —— "这段内存 DMA 方向"(设备读/写)

两者在不同抽象层,服务不同处理者:
  type 给 blk 设备逻辑
  flag 给通用 virtqueue 传输

"读磁盘"时看似重复,实为同一操作在两层的相反投影:
  磁盘读 = 内存写(WRITE)
  磁盘写 = 内存读(无 WRITE)

不省掉是为了:分层解耦(传输层不懂 blk)、DMA 方向控制、交叉校验、复杂请求的逐段精度。
```

**类比**:你网购下单,订单上写"我要买(收货)"(= type,业务意图),快递面单上写"派送方向:送到你家"(= flag,物流方向)。买东西 = 货往你家送,看着重复,但一个是"交易类型",一个是"物流方向",由不同的人(商家系统 vs 快递系统)处理。退货时立刻不同:"退货"(交易)= 货从你家发出(物流反向)。

## 问题 1:那些地址是开机写一次,之后一直复用不变吗?

**分两类,答案不同:**

**(A) 队列结构的地址(desc 表、avail 环、used 环)—— 开机写一次,永久不变。**

这三个地址在初始化时通过 MMIO 写给设备(你代码里的):

```rust

header.set_queue_driver_addr(q.avail_pa);   // avail 环在哪
header.set_queue_device_addr(q.used_pa);    // used 环在哪

```

**写完这几个,设备就永久记住了"这个队列的三个环在哪"。之后所有请求都复用同一个队列、同一组环地址,再也不用通过 MMIO 告诉设备地址了。** 这就是为什么之后每个请求只需要 1 次 notify——地址早就告诉设备了,设备每次自己去那几个固定地址读 avail、写 used。

```rust
desc[1] = { addr: data_pa,   ... }   // 数据地址,写在 desc 里
desc[2] = { addr: status_pa, ... }   // 状态地址,写在 desc 里
```

设备是**读 desc 表**(那个固定地址的环)才知道这次请求的缓冲在哪。所以这些地址**理论上每个请求可以不同**——你可以这次数据缓冲用 0x8050_0200,下次用 0x9000_0000,只要写进 desc 的 addr 就行。

**总结:**

```
队列三环地址(desc/avail/used):MMIO 写一次,永久不变(协议要求复用)

```

所以不是"avail[0] 读 desc[0]、avail[1] 读 desc[1]"那种固定对应——**是 avail.ring[i] 存了一个索引,告诉设备"第 i 个请求的链从 desc[几] 开始"。** 链头是几,你填进 avail.ring 时自己定(填 0、填 3、填 6...)。

## 完整的并发提交流程(修正版)

假设并发 2 个请求(8 槽只够 2 个):

```
1. 请求 A 占 desc[0,1,2]:
   desc[0] = {addr: reqA头, NEXT, next:1}
   desc[1] = {addr: dataA,  NEXT|WRITE, next:2}
   desc[2] = {addr: statusA, WRITE, next:0}

2. 请求 B 占 desc[3,4,5]:
   desc[3] = {addr: reqB头, NEXT, next:4}
   desc[4] = {addr: dataB,  NEXT|WRITE, next:5}
   desc[5] = {addr: statusB, WRITE, next:0}

3. 登记链头到 avail 环:
   avail.ring[0] = 0    ← A 的链头
   avail.ring[1] = 3    ← B 的链头
   avail.idx = 2

4. 一次 notify

5. 设备并发处理 A 和 B,完成谁先放谁:
   (假设 B 先完成)
   used.ring[0] = {id: 3, len: 512}   ← B 完成了(id=3 是 B 的链头)
   used.ring[1] = {id: 0, len: 512}   ← A 完成了(id=0 是 A 的链头)

   读 used.ring[0]:id=3 → "哦 B 完成了"
   读 used.ring[1]:id=0 → "A 完成了"
```

**关键细节:used.ring 里的 `id` 是链头的 desc 索引**,你用它认出"是哪个请求完成了"。B 的链头是 desc[3],所以 B 完成时 used 里 id=3。**完成顺序可能和提交顺序不同**(B 可能先于 A 完成),靠 id 对应回是哪个请求。

## 问题 1:请求头也要 8 个吗?

**对,并发时每个请求要有自己独立的请求头、数据缓冲、状态字节。** 不能共用。

想想为什么:请求 A 读 sector 2,请求 B 读 sector 100。它们的请求头内容不同(sector 字段一个是 2 一个是 100)。如果共用一个请求头,B 会覆盖 A 的 sector——乱套。数据缓冲更是必须分开(A 的数据和 B 的数据不能放同一块,会互相覆盖)。

所以并发 N 个请求,需要:

```
N 个请求头   (各自的 type + sector)
N 个数据缓冲 (各自的 512 字节)
N 个状态字节 (各自的完成码)
```

**这就是为什么真实驱动不是用一个固定 DMA 页,而是动态分配。** 你现在的 `VirtioBlkDma` 是**一个固定页**(一组 req/data/status),所以你**天然只能一次一个请求**——因为只有一份缓冲。想并发,得有 N 份缓冲(动态分配或预分配一个池)。

**你单请求的设计(一份固定 DMA)和你单请求的用法(一次一个)是自洽的。** 要并发就要把"一份 DMA"改成"一池 DMA",这是并发的另一份工作量。**FAT 不需要,跳过。**

## 问题 2:描述符链会超过三个吗?

**会,而且很常见——但不是你想的"分两次读 256"那种。** 你对"分次读脱裤子放屁"的判断是对的,但描述符链变长是为了**完全不同的原因:一次读多个扇区 + 数据分散在不连续的内存**。

### 真相 1:一个请求可以读多个扇区(不止 512 字节)

virtio-blk 一个请求**不限于一个扇区**。你可以一个请求读 8 个扇区(4KB)、64 个扇区(32KB),一次搞定:

```
请求头:type=0, sector=100      "从 sector 100 开始读"
数据缓冲:4096 字节(8 个扇区)   "读 8 个扇区到这"
状态字节
```

设备看请求头 sector=100,看数据缓冲 4096 字节,就知道"读 sector 100~107 共 8 个扇区"。**扇区数 = 数据缓冲长度 / 512,设备自己算。** 不需要你说"读 8 个",数据缓冲多大就读多少。

**所以你担心的"一次 512、想读更多要分次"——不存在。** 一个请求想读多少读多少,数据缓冲开多大,一个请求就读多大。读 32KB?数据缓冲开 32KB,一个请求。

### 真相 2:描述符链变长,是因为大缓冲在内存里不连续

那描述符链什么时候超过 3 个?当**数据缓冲在物理内存里不连续**时。

假设你要读 32KB(64 个扇区),但你的 32KB 缓冲在物理内存里**不是连续的**(分散在好几个物理页,因为内存碎片):

```
缓冲分布:物理页 A(4KB)+ 物理页 B(4KB)+ ... + 物理页 H(4KB)= 32KB,8 块不连续
```

描述符链就要这样:

```
desc[0]: 请求头
desc[1]: 数据段1 → 物理页 A,4KB,WRITE
desc[2]: 数据段2 → 物理页 B,4KB,WRITE
desc[3]: 数据段3 → 物理页 C,4KB,WRITE
...
desc[8]: 数据段8 → 物理页 H,4KB,WRITE
desc[9]: 状态字节
```

**10 个描述符,一个请求。** 这叫 **scatter-gather**(分散-聚集):一个逻辑上连续的读(sector 100~163),数据散落到内存里多个不连续的物理块,每块一个描述符。设备按顺序把磁盘数据 DMA 到这些散落的块里。

**这才是描述符链超过 3 的真正原因:不是分多次读,是一次读的大数据散布在不连续的物理内存,每段内存一个描述符。** 操作还是"一个请求",只是数据缓冲由多段拼成。

### 为什么会有 scatter-gather

因为**用户的 buffer 在虚拟内存里连续,但映射到物理内存可能不连续**。读 32KB 到用户一个连续数组,这数组虚拟地址连续,但背后 8 个物理页可能东一块西一块。设备做 DMA 用的是物理地址,所以要把"虚拟连续、物理分散"的缓冲拆成多段描述符,每段一个物理块。

## VirtioQueue 是什么:三个环的"容器 + 内存布局"

你一直在用的 desc 表、avail 环、used 环,**它们不是三块各自独立 alloc 的内存,而是挤在同一块连续内存里,VirtioQueue 就是管理这块内存的结构。**

你看代码:

```rust
pub struct VirtioQueue {
    pub queue_ppn: PhysPageNum,   // 这块内存的物理页号
    pub queue_pa: usize,          // 物理起始地址
    pub queue_va: usize,          // 虚拟起始地址(内核访问用)
    pub pages: usize,             // 占几页

    pub desc_pa: usize,           // desc 表在这块内存的哪
    pub avail_pa: usize,          // avail 环在哪
    pub used_pa: usize,           // used 环在哪
}
```

**VirtioQueue = "我 alloc 了一块连续内存,desc/avail/used 三个环分别坐落在这块内存的不同偏移处"。** 它就是个**内存布局管理器**。

## 三个环在内存里怎么摆(对照你的常量)

看你文件顶部那些 offset 常量,它们定义了三个环在这块内存里的**位置**:

```rust
const DESC_OFFSET: usize = 0;                              // desc 从头开始
const AVAIL_OFFSET: usize = DESC_OFFSET + DESC_SIZE;       // avail 紧跟 desc 后面
const USED_OFFSET: usize = align_up(AVAIL_OFFSET + AVAIL_SIZE, 4);  // used 再后面(对齐)
```

画出来,这块内存长这样:

```
queue_pa 开始的连续内存:
┌─────────────────────────────────┐ ← queue_pa + 0 (DESC_OFFSET)
│  desc 表                          │
│  desc[0] desc[1] ... desc[7]      │   8 × 16 字节 = 128 字节
├─────────────────────────────────┤ ← queue_pa + DESC_SIZE (AVAIL_OFFSET)
│  avail 环                         │
│  flags, idx, ring[0..8], used_evt │
├─────────────────────────────────┤ ← USED_OFFSET (对齐后)
│  used 环                          │
│  flags, idx, ring[0..8], avail_evt│
└─────────────────────────────────┘
```

**三个环是同一块内存的三个区段。** `desc_pa = queue_pa + 0`、`avail_pa = queue_pa + DESC_SIZE`、`used_pa = queue_pa + USED_OFFSET`。这就是你 `new_with_layout` 里算的:

```rust
desc_pa: pa + DESC_OFFSET,
avail_pa: pa + AVAIL_OFFSET,
used_pa: pa + used_offset,
```

## 为什么它是"核心"

我说 virtqueue 是 virtio 的心脏——现在你能看清为什么了:

**1. 它是 guest 和设备唯一的"约会地点"。** 整个通信就发生在这块内存上:

- 你往 desc 写请求结构、往 avail 写"准备好了"
- 设备从 desc/avail 读请求、往 used 写"完成了"
- 你从 used 读完成结果

**这块内存(VirtioQueue 管的)就是 guest 和设备的共享白板。** 所有交互都在这上面。MMIO 寄存器只是用来"开机告诉设备这块白板在哪"(`set_queue_desc_addr` 等)和"按铃"(notify)。**真正的协作全在 VirtioQueue 这块内存。**

**2. 它的地址,开机时通过 MMIO 写给设备,之后设备永久知道。** 回到你之前的问题——

```rust
header.set_queue_desc_addr(q.desc_pa);     // 把 desc_pa 告诉设备
header.set_queue_driver_addr(q.avail_pa);  // avail_pa
header.set_queue_device_addr(q.used_pa);   // used_pa
```

**这就是"把白板的位置告诉设备"。** 设备记住后,每次自己去 `desc_pa` 读描述符、去 `avail_pa` 读你提交了啥、往 `used_pa` 写完成。VirtioQueue 的三个 `_pa` 字段,就是为了开机时报给设备的。

**3. 它对 mmio 和 pci 完全一样。** 我说"第 2 层 virtqueue,mmio 和 pci 没区别"——因为 VirtioQueue 这块共享内存的结构(三个环的布局)是 virtio 规范定死的,和"用 mmio 还是 pci 找设备"无关。riscv 和 loongarch 的 VirtioQueue 内存布局一模一样,区别只在怎么把地址告诉设备(mmio 写寄存器 vs pci 写配置空间)。

## 你代码里 VirtioQueue 的两个方法

```rust
pub fn new_legacy() -> Option<Self>   // legacy 版的布局(used 环要页对齐)
pub fn new_modern() -> Option<Self>   // modern 版的布局(used 环 4 字节对齐就行)
```

为什么两个?**legacy 和 modern 版 virtio 对 used 环的对齐要求不同:**

```rust
const LEGACY_USED_OFFSET = align_up(AVAIL_OFFSET + AVAIL_SIZE, PAGE_SIZE);  // 页对齐(4KB)
const MODERN_USED_OFFSET = align_up(AVAIL_OFFSET + AVAIL_SIZE, 4);          // 4字节对齐
```

legacy 要求 used 环从一个新页开始(老协议的限制),所以 legacy 布局占 2 页、used 在第二页;modern 宽松,三个环挤在 1 页就行。这就是 `LEGACY_QUEUE_PAGES = 2` vs `MODERN_QUEUE_PAGES = 1` 的来源。**纯粹是对齐要求不同导致的布局差异**,环的逻辑结构一样。

## 那些 `_va` 方法:你怎么访问这三个环

```rust
fn desc_va(&self) -> usize  { self.queue_va + DESC_OFFSET }
fn avail_va(&self) -> usize { self.queue_va + AVAIL_OFFSET }
fn used_va(&self) -> usize  { self.queue_va + (self.used_pa - self.queue_pa) }
```

设备用**物理地址**(DMA 用 pa),但**你的内核代码访问内存要用虚拟地址**(va)。所以 VirtioQueue 同时存了 pa(给设备)和 va(给你自己)。你往 desc 写、读 used,都通过 va:

```rust
unsafe fn desc_mut(&self, index) -> *mut VirtqDesc {
    (self.desc_va() as *mut VirtqDesc).add(index)   // 通过 va 访问 desc[index]
}
```

**同一块内存,两个地址视角:设备用 pa 做 DMA,你用 va 读写。** VirtioQueue 把两个都存着。

## 串起来:VirtioQueue 在整个流程里的位置

```
初始化:
  VirtioQueue::new_legacy()  → alloc 一块连续内存,算好三个环的 pa/va
  把 desc_pa/avail_pa/used_pa 通过 MMIO 写给设备  → 设备记住白板位置

每个请求:
  你用 desc_va 往 desc 表写描述符链      (写白板的 desc 区)
  你用 avail_va 往 avail 环登记链头      (写白板的 avail 区)
  notify 设备
  设备用 desc_pa/avail_pa 读你写的       (设备读白板)
  设备 DMA 数据 + 用 used_pa 写完成      (设备写白板的 used 区)
  你用 used_va 读 used 环收割            (读白板的 used 区)
```

**VirtioQueue 就是这块"白板"的管理者。** 它不参与每个请求的逻辑(那是 desc/avail/used 的内容),但它**承载**了这一切——三个环都活在它管的内存里。

## 为什么我前面没单独提它

因为我一直在讲三个环的**内容和行为**(desc 存什么、avail 怎么用、used 怎么收割),而 VirtioQueue 是**装这三个环的盒子**。你理解了三个环,其实就理解了 VirtioQueue 的全部内涵——它只是"把三个环放在一块连续内存里,记好各自偏移和 pa/va"的容器。**没有额外的魔法,就是内存布局 + 地址记账。**

```
VirtioQueue = 一块连续内存 + 三个环在里面的布局(desc/avail/used 各占一段)

```
